## 1. 60 卡集群规划(001–015)

设计三原则:**反亲和(每能力 ≥2 物理节点,单节点挂了功能仍在线只降并发)**、**4 卡模型整节点独占**、**1 卡 offload 模型按主机内存红线打包**。

| 节点 | 角色 | 每节点布局(4 卡) | 产出实例 |
|---|---|---|---|
| 001 002 | 视频 t2v | 1× wan2.2-t2v(4 卡) | 2 |
| 003 004 | 视频 i2v | 1× wan2.2-i2v(4 卡) | 2 |
| 005 006 | 数字人 s2v-480p | 1× infinitetalk-480p(4 卡) | 2 |
| 007 008 | 数字人 s2v-720p | 1× infinitetalk-720p(4 卡) | 2 |
| 009 010 | 视频后期 | 2× vace + 2× sr(GPU0/1=vace,GPU2/3=sr) | 4 vace,4 sr |
| 011–015 | 图片+语音 | 1× qwen-edit(GPU0)+ 1× z-image(GPU1)+ 2× tts(GPU2/3) | 5 edit,5 t2i,10 tts |

**GPUStack 表达方式**:同一能力 = **一个 Model + Replicas=N**(名字统一 + 自动 HA + least-pending 负载均衡),而不是建 N 个 `-1/-2` 分开的模型。Manual 调度选 N×(卡/副本)张卡,GPUStack 自动把副本铺到对应节点。

### 各模型部署配置速查

统一项:Source=Local Path、Cluster=`a100-image-video`、Scheduling=Manual、Version=Auto、**Enable Model Route 勾上**、**4 卡模型取消 Allow Distributed Inference**(单节点 in-process torchrun,不能跨 worker 拆)。Model Path 是**完整绝对路径** `/nfs-models/wuhanjisuan894/models/...`。

| Model | Model Path(前缀省略 `/nfs-models/wuhanjisuan894/models/`) | Backend | Category | Repl × GPU | Backend Parameters |
|---|---|---|---|---|---|
| `wan2.2-t2v` | `Wan-AI/Wan2.2-T2V-A14B` | LightX2V | Video | 2×4 | 无 |
| `wan2.2-i2v` | `Wan-AI/Wan2.2-T2V-A14B`(同 t2v 基座) | LightX2V | Video | 2×4 | `--model-cls wan2.2_moe_distill` `--task i2v` |
| `infinitetalk-480p` | `Wan2.1-I2V-14B-480P` | LightX2V | Video | 2×4 | `--model-cls infinitetalk` `--profile infinitetalk-480p/int8-4card` |
| `infinitetalk-720p` | `Wan2.1-I2V-14B-480P`(基座与 480p 相同) | LightX2V | Video | 2×4 | `--model-cls infinitetalk` `--profile infinitetalk-720p/int8-4card` |
| `wan2.2-vace` | `Wan2.2-VACE-Fun-A14B` | LightX2V | Video | 4×1 | 无(路径含 vace 自动选 wan2.2_moe_vace) |
| `seedvr2` | `ByteDance-Seed/SeedVR2-3B` | LightX2V | Video | 4×1 | `--model-cls seedvr2`(引擎按 `<Model Path>/seedvr2_ema_3b.pth` 加载) |
| `qwen-image-edit` | `Qwen-Image-Edit-2511` | LightX2V | Image | 5×1 | `--task i2i` |
| `z-image` | `Z-Image-Turbo` | LightX2V | Image | 5×1 | 无 |
| `indextts-2` | `IndexTTS-2` | **IndexTTS** | 自动 TTS | 10×1 | 无 |

**关键选型记忆**:
- **InfiniteTalk 480p/720p 的 Model Path 都填 `Wan2.1-I2V-14B-480P`**——它是基座(VAE/T5/CLIP,分辨率无关),分辨率由 `--profile` → config → 不同 int8 DiT(`Wan2.1-Distill-Models/wan2.1_i2v_{480,720}p_int8_...`)决定。NFS 上没有单独的 720p 基座目录。
- **s2v 的引擎是 InfiniteTalk,不是 Wan2.2-S2V**(后者 40 步无蒸馏 22min/条,只做画质基线;InfiniteTalk 4 步蒸馏,480p 54s / 720p 137s per 5s clip)。720p 音频上限 ≤25-30s,480p ≤60-75s。
- **vace 只能单卡**:4 步 LoRA 720p 形态 4 卡常驻 VAE 前处理 OOM(实测"三审死刑");单卡 offload 是唯一活路(344s/条)。多卡只对 40 步无蒸馏形态有意义,但那是慢速创作档不上量产。靠"2/节点 × 副本"拿并发。
- **主机内存红线(调度器看不见,部署期人肉控)**:qwen ≤2/节点(~60G shmem/实例)、vace ≤2/节点(~55G)、009/010 的 2vace+2sr ≈194G/251G 贴线;offload 实例 ≤3/节点、错峰启动 ≥2min。

## 2. 部署动线(从出包到铺满)

### 2.1 出新包(gpustack overlay)
GitHub Actions 手动触发 **Pack GPUStack overlay to ACR**(`pack-acr-overlay.yml`,workflow_dispatch,默认 `dest_tag=lx2v-dev`、`ui_ref=main`)。约 10min 出多架构(amd64+arm64)包,推 ACR,tag = 浮动 `lx2v-dev` + 不可变 `lx2v-YYYYMMDD-HHMM-<sha8>`。

> 引擎镜像(lightx2v/indextts2)是**各自独立仓的独立 CI**,和 gpustack overlay 无关。lightx2v 的 s2v/sr/vace 支持随 `516b9306` build 进 `cbebe326`,不需要为它重打。

### 2.2 238 侧 prepare-transfer(刷 NFS tar + 同步脚本)
```bash
# 238 上(先更新 /root/lx2v-node.sh 到最新版)
bash /root/lx2v-node.sh prepare-transfer
```
拉三镜像(gpustack/lightx2v/indextts2 arm64)→ save 成 NFS `_transfer/` tar(gpustack 4.4G、lightx2v 9.8G、indextts2 11G)→ 顺带把 `/root/lx2v-node.sh` 同步到 NFS。**坑见 §3.4(多架构 docker save)。**

### 2.3 238 server 手动升级(x86,不用脚本)
```bash
docker pull crpi-.../reputationly/gpustack:lx2v-20260712-0638-e3ace645
docker stop gpustack-server && docker rm gpustack-server
docker run -d --name gpustack-server --restart unless-stopped -p 80:80 \
  --volume gpustack-data:/var/lib/gpustack \
  --volume /nfs-output:/nfs-output \
  crpi-.../reputationly/gpustack:lx2v-20260712-0638-e3ace645 \
  --system-default-container-registry quay.io
curl -s -o /dev/null -w '%{http_code}\n' http://localhost/   # 200=就绪
```
升级前先 `docker inspect gpustack-server --format '{{json .Mounts}}'` 确认是**命名卷 `gpustack-data`**(不是匿名 hash 卷,否则"数据消失"坑)。

### 2.4 15 台 worker 一键升级(fleet 脚本,本次新增)
```bash
# 238 上,一条命令批量升所有 worker
bash /root/lx2v-fleet.sh upgrade-gpustack
# 输出 ✅ <ip> 逐台 + 末尾 OK=15 FAIL=0
```
详见 §4。每台自动继承旧 worker 的 token/卷/IP/server-url,重建 worker 带上新镜像 + `-v /nfs-data`(见 §3.1),期间引擎实例容器不受影响。

### 2.5 UI 逐个部署模型
按 §1 的 9 个模型清单铺。顺序建议:先 4 卡视频(t2v/i2v/s2v),再 vace/sr(**错峰**),最后图片+语音(qwen/z-image/tts)。图片+语音节点(011-015)和视频后期节点(009/010)**独立,可并行部署**。

验收:每个实例容器日志出 launcher 选型行(`model_cls=… profile=…`)+ `/ready 200`。带输入文件的能力走 new-api 体验区冒烟(门面直连要手工物化输入,繁琐)。

## 3. 踩坑全记录(按重要度)

### 3.1 【最关键】s2v/vace 的 `/nfs-data` 路径 + 镜像化部署挂载
**现象**:s2v/vace 实例起来报 `No such file /nfs-data/models/.../wan2.1_i2v_480p_int8...safetensors`。
**根因**:LightX2V 里 s2v(InfiniteTalk)/vace 的 config JSON **写死 `/nfs-data/models[-int8]/...`**(从 LightX2V dev 环境带来),而集群 NFS 是 `/nfs-models/wuhanjisuan894/...`。t2v/qwen 等 config 用的是对的 `/nfs-models/...`,只有 s2v/vace 这批新 config 用 `/nfs-data/`。
**桥接**:节点宿主机有软链 `/nfs-data -> /nfs-models/wuhanjisuan894`(`ensure_nfs` 建)。
**坑中坑**:先以为把 `/nfs-data` 加进 `GPUSTACK_EXTRA_MOUNTS`(env)就行——**实测不生效**。因为集群启用 **镜像化部署(`GPUSTACK_RUNTIME_DEPLOY_MIRRORED_DEPLOYMENT`)**:推理容器的挂载**镜像 worker 容器自己的 `-v` 卷**,不走 `_get_extra_mounts` 的 EXTRA_MOUNTS。而 `run_worker` 只 `-v /nfs-models -v /nfs-output`,没有 /nfs-data。
**修法**:给 **worker 容器**加 `--volume /nfs-data:/nfs-data`(docker 解析宿主软链、挂目标到容器同路径),镜像化部署会传给推理容器。`lx2v-node.sh` 已改:`run_worker` 加 `-v /nfs-data`;upgrade 里 `ensure_extra_mount`(EXTRA_MOUNTS 也补,文档路径完整性)+ `ensure_symlink /nfs-data` 兜底。改完各节点 `upgrade-gpustack`,**再到 UI 重新部署实例**(worker 重启不给旧实例换挂载)。验证:`docker inspect gpustack-worker --format '{{json .Mounts}}' | grep /nfs-data`。

### 3.2 video category 枚举漂移 bug(3 处)
fork 给 `CategoryEnum` 加了 `VIDEO`,但多处硬编码列表没跟上(同 [`acr-overlay-copy-list-gotcha`] 白名单坑):
- `schemas/model_routes.py` `validate_categories` 白名单漏 `video` → video 类模型建 route 后 **`GET /model-routes` 校验崩、Routes 页打不开**。
- `worker/runtime_metrics_aggregator.py` 跳过 metrics 抓取漏 video → video 实例反复 scrape 不存在的 `/metrics` 刷日志。
- `routes/benchmarks.py` 拒绝 benchmark 判断漏 video。
**修法**:三处补 `video`/`is_video_model()`,并**把这 3 个文件加进 `pack/Dockerfile.acr` 的 COPY 白名单**(不加则改了随主镜像静默漏),出新包 `e3ace645`,fleet 重升。用 agent 全仓扫过,其余命中(`is_omni_model`/GMU)是 VLLM 专属路径、video 不进,排除。

### 3.3 Enable Model Route 只能建时设
UI 里 **Enable Model Route 是创建时才能设的,建好之后编辑改不了**(要删模型重建)。而且负载均衡**不依赖它**——异步视频/语音走门面 `least-pending`(`routes/videos.py` 直接按 model.id 取 running 实例);Model Route 只对"走 OpenAI 网关 sync 路径"有意义。**结论:建时统一勾上**(bug 已修,勾了无害且多留网关路径,免将来删重建)。

### 3.4 多架构 docker save(containerd 存储)
**现象**:238 上 prepare-transfer 的 `docker save gpustack:lx2v-dev` 报 `unable to create manifests file: NotFound: content digest ... not found`。
**根因**:新包是 amd64+arm64 多架构 manifest;238 docker 用 containerd 镜像存储,`docker save <多架构tag>` 会尝试导出整个 manifest list(含只拉了 arm64、amd64 未拉的那份)。
**立刻解**:先 `docker pull --platform linux/amd64 <tag>` 让两平台都在本地,再 save。
**根治**:`docker save --platform linux/arm64 <tag>`(只导 arm64)。`lx2v-node.sh` 的 gpustack save 已加 `--platform linux/arm64`。

### 3.5 offload 模型冷启慢 / 主机内存挤爆 page cache / 错峰
**现象**:vace 加载 172s vs 566s 两台差异巨大,慢的卡在 `Loading ...models_t5_umt5-xxl-enc-bf16.pth`(11G T5)读了近 8min;且某节点(10 号)**每次都慢**。
**根因**:vace offload 权重(~55-75G/实例)常驻主机内存,把之前实例读过、缓存的 11G T5 **从 page cache 挤出去** → 新实例只能从 NFS 冷读;同一节点多实例同时冷启 → NFS 读争用(报告实测"3×T5 并发读 18min")。个别节点(10)持续慢则指向该节点 NFS/网络更慢或空闲内存更少,需 `dd` 冷读带宽 + `MemAvailable` + `ping NFS server` 对比诊断。
**规避**:vace/sr **一张卡一张卡起、每步等 `/ready 200`(page cache 稳、NFS 空)再加下一个**,盯 `MemAvailable ≥15G`;offload 模型别往慢节点堆。这些是一次性冷加载成本,加载完常驻、跑推理不受影响。

### 3.6 GitHub Actions 老报错(CI + Pack)
- **CI(`ci.yml`)**:`make lint`(pre-commit)挂在 **flake8 复杂度 + black**。真身只两条:`scheduler.py:evaluate_pretrained_config` 复杂度 16>15(fork 加内置后端分支撑上去的)+ `indextts.py` 待 black 重排。修:抽 `_evaluate_builtin_backend_config` helper 降到 15、black 重排。**注意**:CI 的 flake8 hook 独立环境**没装 flake8-bugbear**,满屏 B950/B006 是裸 `flake8` 的过度报告,不进 CI。
- **Pack(`pack.yaml`)**:上游全量出包,缺 registry 密码报 `Password required`、fork 又不用(用 `pack-acr-overlay.yml`)。修:**关掉它的 push/PR 触发**,只留 workflow_dispatch。
- **本地防线**:装了 pre-commit(commit 时)+ pre-push(`make lint`,失败拦 push)git 钩子,push 前必过一遍 CI 同款 lint。

## 4. lx2v-fleet.sh —— manager 批量编排(本次新增)

**动机**:15 台逐个 ssh + 手动 `cp + upgrade-gpustack` 太慢;一条命令扇出到全体。

**一次性准备**:238 免密到各节点 + 节点清单
```bash
ssh-keygen -t ed25519 -N '' -f ~/.ssh/id_ed25519            # 若没有
# 从 GPUStack 拉 worker IP 生成清单(排除 manager 自身)
curl -s "http://localhost/v1/workers?perPage=100" -H "Authorization: Bearer $KEY" \
  | python3 -c "import sys,json;[print(w['ip']) for w in json.load(sys.stdin)['items'] if w['ip']!='10.0.0.238']" \
  | sort -u > /root/lx2v-nodes.txt                          # 一行一个内网 IP
for ip in $(cat /root/lx2v-nodes.txt); do ssh-copy-id -o StrictHostKeyChecking=accept-new root@$ip; done
```
> `/root/lx2v-nodes.txt` 含真实内网 IP,**已加 .gitignore 不入库**。`AGENTS.md` 同理。

**用法**:把 `lx2v-node.sh` 的任意子命令透传给全体
```bash
bash /root/lx2v-fleet.sh upgrade-gpustack                          # 全体升 worker 镜像 + 补挂载
bash /root/lx2v-fleet.sh -j 3 upgrade-engine --engine lightx2v --offline   # 大 tar 降并发防 NFS 抢
bash /root/lx2v-fleet.sh upgrade-engine --engine indextts --offline
bash /root/lx2v-fleet.sh status                                    # 全体巡检
```
机制:读清单 → 并发(默认 5)ssh 每台 `cp /nfs-models/_transfer/lx2v-node.sh /root/ && bash /root/lx2v-node.sh <args>` → 每台日志 `/tmp/lx2v-fleet/<ip>.log` → 末尾汇总 `OK=/FAIL=`。选项 `-j N`(并发)、`-f FILE`(换清单)、`--seq`(串行)。**engine --offline 建议 `-j 3`**(15 台同时从 NFS load 10G tar 会抢带宽)。运行前会把 manager 的 `/root/lx2v-node.sh` 同步到 NFS,保证各节点拿最新版。

**本次实测**:15 台 `upgrade-gpustack` 首跑 `OK=15 FAIL=0`,单台 ~45s-1min。
