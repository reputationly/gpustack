## 3. 升级:238 server + 163 worker

### 3.1 238(含两次事故与处置,均已修复根因)

**操作**:`docker tag <旧image> gpustack-rollback:pre-m4`(回滚锚)→ pull `lx2v-dev` → stop/rm → 原参数重启(`-p 80:80 -v gpustack-data:/var/lib/gpustack -v /nfs-output:/nfs-output … --system-default-container-registry quay.io`)。

**事故一:迁移链断裂**(`lx2v-…-0750` 镜像)。启动报:

```
Database migration failed: 'c4d7e8f9a0b1'  (KeyError)
```

根因:我们迁移的 `down_revision` 指向 fork 链上一版 `c4d7e8f9a0b1`,而官方 base 镜像只带发布版迁移链、没有该中间版本。**不能**把 upstream 中间迁移塞进镜像(对应代码不在,会把库改坏);修法 = Dockerfile 构建期探测 base 实际链头(`b2c3d4e5f6a7`)并 sed 改写 `down_revision` + 断言链头。链解析阶段就失败、未执行任何 SQL,**DB 无损**,换 `0809` 镜像重启即过:

```
Running database migration. → Database migration completed.   (0.9s)
Init built-in backend LightX2V in database
```

**事故二:"数据消失"(实为匿名卷)**。新容器起来后 `models`/`workers` 全空、登录页要初始密码。排查:

```bash
docker volume ls        # gpustack-data 创建时间 = 当天 16:05(新的空卷!)
ls /var/lib/docker/volumes/a5311fd7…/_data/   # 7月5日的老数据在这个匿名卷里
```

根因:**Phase A 首装时 server 的 docker run 没带 `--volume`**,数据落在镜像 `VOLUME /var/lib/gpustack` 自动生成的匿名卷;本次按 runbook 用命名卷 → 挂了个空卷初始化了新库。`docker rm` 不删卷,老数据完好。恢复:

```bash
docker stop gpustack-server && docker rm gpustack-server
sudo rm -rf /var/lib/docker/volumes/gpustack-data/_data/*
sudo cp -a /var/lib/docker/volumes/a5311fd7…/_data/. /var/lib/docker/volumes/gpustack-data/_data/
# 原命令重启 → 老数据 + 存量库增量迁移,全部回来
```

**存量库迁移验证**(本轮最关键修复的真机证据):

```bash
docker exec -u postgres gpustack-server psql -d gpustack -c \
  "SELECT unnest(enum_range(NULL::videotaskstateenum));"
# QUEUED / ASSIGNED / RUNNING / DONE / FAILED / CANCELED  (6 rows) ✅
docker exec -u postgres gpustack-server psql -d gpustack -c "SELECT name, backend FROM models;"
# z-image | LightX2V ✅(老数据无损)
```

**经验**:升级任何容器前先 `docker inspect <容器> --format '{{json .Mounts}}'` 看真实卷名;老匿名卷保留数日作备份后再回收。
